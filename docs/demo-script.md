# Demo 脚本 — AuditDraft AI 评委演示流程

> 路演用「一镜到底」脚本：从登录到导出底稿，配套话术与亮点提示。
> 目标时长 **3–5 分钟**，核心卖点：**混合推理（规则确定性 + LLM 生成）** 与 **可解释审计**。

## 0. 演示前置检查（开场前 5 分钟完成）

- [ ] `.env.local` 已配置真实 Supabase / OpenAI（或 Anthropic）/ Pinecone key。
- [ ] 已执行 `supabase/migrations/*.sql`（建表 + RLS + storage bucket）。
- [ ] 已 `npm run seed:rag` 灌入 RAG 知识库（15 条：准则 / 风险规则 / 程序 / 案例）。
- [ ] 已 `npm run check:env` 通过（环境变量齐全）。
- [ ] 已准备好演示账号（建议用**全新账号**，证明无本地脏数据依赖）。
- [ ] 演示数据集就位：`samples/expense_transactions.csv`。
- [ ] 应用已启动：本地 `npm run dev` 或线上 Vercel URL。

## 1. 演示数据集与预期结果（评委最关心的「确定性」）

数据集 `samples/expense_transactions.csv`（11 笔交易，审批阈值 `10000`）刻意埋入 **4 类风险**：

| 行 | 供应商 | 金额 | 审批人 | 发票号 | 命中风险 | 等级 |
| --- | --- | --- | --- | --- | --- | --- |
| 1、2 | ABC Corp | 5000 | 无 | INV-1001 | **重复付款 / 重复发票**（同供应商+同金额+同发票号） | high |
| 4 | LMN Inc | 20000 | 无 | INV-3001 | **无审批大额**（>10000 且无审批人） | high |
| 5、6 | QuickBuy | 9000 / 9500 | Alice | INV-4001/2 | **拆分报销**（≥2 笔贴边低于阈值 [8000,10000)） | medium |
| 11 | Mega Supplier | 500000 | Carol | INV-6001 | **金额离群**（z-score 远超 2） | medium |

> 整体风险等级 = `HIGH`（取所有 finding 的最高 severity）。
> 这些结论来自**确定性规则引擎**（`src/lib/rules`），不依赖 LLM —— 同样的输入永远得到同样的结论，**可复现、可审计**。

## 2. 现场演示流程（含话术）

### Step 1 · 登录（10 秒）
- 操作：用全新账号登录 `/login`。
- 话术：「AuditDraft 是 SaaS 级多租户系统，数据按用户隔离，依赖 Supabase 行级安全（RLS）。」

### Step 2 · 上传 Excel/CSV（15 秒）
- 操作：在 `/dashboard` 上传 `expense_transactions.csv`，系统自动创建任务并触发审计。
- 话术：「传统审计：会计师手工翻这张表要 2–4 小时。我们的目标是 **30 秒**。上传后自动进入 AI 审计流程。」

### Step 3 · 观察工作流实时推进（20 秒）—— **亮点 1：流程编排**
- 操作：指向 Workflow Tracker 的 7 个节点依次点亮。
- 话术：「整个审计被编排成一张 LangGraph 工作流图：
  `解析数据 → AI 审计计划 → 规则引擎 → 异常检测 → 风险评级 → 底稿生成 → 报告导出`。
  其中**规则引擎是确定性的，绝不依赖大模型**；大模型只负责理解与表达。这就是我们的**混合推理**架构。」

### Step 4 · 查看风险卡片（30 秒）—— **亮点 2：风险识别**
- 操作：展示按 severity 着色的 Risk Cards（红=high / 橙=medium）。
- 话术：「系统自动识别出 4 类风险：重复付款、无审批大额、拆分报销、金额离群。
  红色是高风险——比如 ABC Corp 这两笔金额、发票号完全相同，是典型的重复付款舞弊信号。」

### Step 5 · 展开可解释面板（40 秒）—— **亮点 3：可解释性（核心差异点）**
- 操作：展开任一 Risk Card 的 Explainability Panel。
- 话术：「这是我们和『黑盒 AI』最大的区别。每一条结论都附带四要素：
  - **触发规则**：`duplicate_payment: same vendor + same amount + same invoiceId`
  - **数据证据**：命中的原始交易（行号、金额、发票号）
  - **审计准则引用**：《中国注册会计师审计准则第 1141 号》（来自 RAG 知识库）
  - **自然语言解释**：为什么这构成风险。
  **没有证据，就没有结论** —— 这是可归档、经得起复核的审计底稿。」

### Step 6 · 阅读审计底稿（30 秒）—— **亮点 4：生成式输出**
- 操作：滚动 Audit Report Viewer（Notion 风格可折叠分节）。
- 话术：「大模型基于 RAG 检索到的审计准则，把这些确定性发现整合成一份标准审计工作底稿：
  审计概述 → 审计程序 → 风险事项 → 风险评级与内控评价 → 结论与建议。」

### Step 7 · 一键导出 Word / PDF（15 秒）
- 操作：点击导出按钮下载 Word（或 PDF）。
- 话术：「一键导出可直接归档的 Word/PDF 底稿。Excel 进，标准底稿出，**全程 30 秒**。」

## 3. 关键卖点总结（收尾 30 秒）

1. **效率**：2–4 小时 → 30 秒。
2. **可靠**：确定性规则引擎做硬检测，结论可复现、可审计。
3. **可解释**：每条结论 = 触发规则 + 数据证据 + 准则引用 + 解释。
4. **可扩展**：LLM / RAG / 规则 / DB / UI 全部面向接口解耦，可独立替换。

## 4. 评委可能追问 & 应答

| 提问 | 应答要点 |
| --- | --- |
| 「AI 会不会乱编风险？」 | 不会。确定性结论由规则引擎产出；prompt 中硬约束 LLM「不得新增无证据结论」（见 `EXPLAINABILITY_GUARD`），LLM 只做解读与表达。 |
| 「换个大模型行不行？」 | 行。`LLM_PROVIDER=openai\|anthropic` 一键切换，业务层只依赖 `LLMProvider` 接口。 |
| 「数据安全吗？」 | 多租户按 `user_id` 隔离，Supabase RLS 强制；service role key 仅服务端使用，不进客户端 bundle。 |
| 「规则不够怎么办？」 | 规则是独立纯函数，注册到 `RULE_REGISTRY` 即可新增，无需改动工作流。 |
| 「为什么用 LangGraph？」 | 审计流程是一张可编排的有向图，节点职责清晰、可单测、可插拔；流程顺序集中在图定义，不散落在业务代码里。 |

## 5. 故障预案（演示翻车时）

- **Live key/网络异常**：改用 `npm run test`（17 项全绿）现场展示规则引擎与图编排的确定性单测结果，证明核心逻辑无需联网即可复现。
- **审计超时**：强调规则引擎结论已秒出；LLM 文本生成可重试（任务状态机支持「重试」按钮）。
- **上传失败**：直接打开一条历史 `done` 任务展示报告与导出。
