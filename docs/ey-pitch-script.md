# AuditDraft · 安永华明 AI 赛路演讲稿

> **定位一句话（全场统一）**  
> AuditDraft 不是「用大模型写底稿」，而是安永可用的 **费用与报销循环 AI 协审工作台**：确定性规则产出可复现发现 → RAG 锚定准则 → LLM 只做理解与成文 → 每条结论四要素可解释 → 输出进入 Prepare → Review → Archive。
>
> **目标时长**：路演 PPT **5–6 分钟** + Demo **3–5 分钟**（或 PPT 3 分钟压缩版见文末）。  
> **红线**：不说「替代审计师 / 自动出审计意见 / 100% 发现舞弊 / 颠覆审计」。

配套文件：
- PPT：[`docs/pitch/AuditDraft-EY-Pitch.pptx`](./pitch/AuditDraft-EY-Pitch.pptx)
- Demo 操作脚本：[`docs/demo-script-ey.md`](./demo-script-ey.md)

---

## 开场前 30 秒（上场前默念）

1. 评委尺子是「能否进项目组 / 过质量复核 / 过信息安全」，不是黑客松。  
2. 先抛结论，再给证据。  
3. 每页只打一个点；Demo 负责「一眼可信」。

---

## 第 1 页 · 封面（15s）

**屏上**：AuditDraft AI · 费用循环 AI 协审工作台  
**说**：

> 各位评委好。我们是 AuditDraft。  
> 今天只讲一件事：怎样把费用与报销循环的初筛，做成**可复核、可评测、可治理**的协审能力——而不是又一个聊天写底稿的 Demo。

---

## 第 2 页 · 问题（40s）

**说**：

> 费用循环的现实是：明细动辄上千行，助理靠抽查翻表；重复付款、无审批大额、拆分报销、金额离群——这些红旗方法上清楚，但**规模化执行贵、漏检贵、复核贵**。  
> 市场上的「贴表问 LLM」更快，却带来三个致命问题：结论不可复现、证据链进不了底稿、幻觉风险过不了质量复核。  
> 事务所真正要的不是更会写的模型，而是**进 Prepare–Review 链条的工作台**。

**停顿点**：让评委点头——问题是他们自己的痛。

---

## 第 3 页 · 解法定位（35s）

**说**：

> 所以 AuditDraft 的边界写得很清楚：  
> **全量初筛 + 可复核草稿底稿**，不替代注册会计师判断，不直接出审计意见。  
> 架构上三条硬线：  
> 一，硬结论只来自确定性规则引擎，不依赖大模型；  
> 二，RAG 把准则与方法锚进上下文；  
> 三，LLM 只负责理解与成文。  
> 一句话：**规则保确定性，模型做表达，人做判断。**

---

## 第 4 页 · 混合推理架构（45s）

**说**：

> 编排用 LangGraph 七步：解析 → 计划 → **规则引擎** → 异常 → 风险 → 底稿 → 导出。  
> 为什么用图而不是脚本？因为审计是质量流程：节点可测、可降级、可插人工门闩——像项目组工作方式，不像一次扫表。  
> 技术上 LLM / RAG / 规则 / 数据库全部接口解耦，provider 可替换；业务上同一套引擎可扩展到 AP、Payroll、收入截止——今天演示费用循环，方法论可复制。

---

## 第 5 页 · 可解释四要素（50s）——核心差异化

**说**：

> 评委最关心：结论能不能上复核底稿？  
> 我们的契约是：**没有证据，就没有结论。**  
> 每一条风险必须同时给出四要素——触发规则、数据证据、准则引用、风险解释。  
> 这不是产品文案，是系统约束：规则层产出结构化 finding，可解释性校验进 Golden Set，不过关就不过线。  
> 所以输出进的是工作底稿语言，不是聊天窗口截图。

**手势**：指四要素卡片，语气放慢。

---

## 第 6 页 · 证据包：评测数字（45s）——技术评委

**说**：

> 空口说可靠不够。我们有离线 Golden Set：同一演示集 **135** 个标注 case，规则引擎离线回归——  
> **Precision 1.0 · Recall 1.0 · F1 1.0 · 可解释性 100% 通过**，且**不依赖大模型**。  
> 同一张表：人工抽查基线约 **45 分钟、命中 24、漏报 4、误报 6**；系统全量初筛约 **8 秒级、埋雷 28/28 命中、误报 0**。  
> 实用性不是口号，是对照实验。

> 注：现场以 `/eval` 页实时数字为准；若口径更新，以 `evals/out/latest.json` 与 `evals/human-baseline.json` 为准。

---

## 第 7 页 · 为什么不是 ChatGPT 贴表（40s）

**说**：

> 对照很简单。纯 LLM 扫表：快，但可能编造风险、无证据、不可复现。  
> AuditDraft：规则结论 100% 可复现；LLM 换掉，硬发现不变；还可规则-only 降级——模型额度没了，初筛与底稿骨架照样出。  
> 对事务所来说，这叫**可治理**，不是更会聊天。

---

## 第 8 页 · 负责任 AI 与合规（40s）——信安/质量评委

**说**：

> 安永尺子还有第三条：过得了信息安全与质量。  
> 演示数据全脱敏合成；生产按用户 RLS 隔离；service role 仅服务端。  
> 高风险必须人工复核状态闭环，才能进最终底稿包叙事。  
> 全链路版本：规则集 / Prompt / 知识库写入报告元数据；Kill switch 与降级模式写进系统卡。  
> 明确禁止用途：**不得在无复核情况下对外披露发现，不得直接出具审计意见。**

---

## 第 9 页 · 现场演示路径（25s）——过渡到 Demo

**说**：

> 接下来 3 分钟现场跑通：上传约 3000 行费用明细 → 规则全量初筛 → 展开四要素 → `/eval` 甩数字 → 底稿标注 AI-assisted draft。  
> 若模型不可用，我们走规则-only——这是设计能力，不是事故。

**动作**：切 Demo，按 `demo-script-ey.md` 执行。

---

## 第 10 页 · 试点与推广（35s）——商业/创新评委

**说**：

> 落地不谈颠覆，谈试点。  
> 第一期：费用/报销循环初筛，服务一个项目组；成功指标是初筛时效、召回、High 风险复核闭环率。  
> 过质量复核与工具认证后，同一套 graph + 规则包，复制到应付、薪酬、收入截止。  
> 我们要进的是**项目组工具箱**，不是年度创新展台。

---

## 第 11 页 · 收尾三句话（30s）

**说**：

> 总结三点：  
> **效率**——全量初筛从小时级翻表到秒级/分钟级；  
> **可靠**——规则确定性 + 离线评测数字；  
> **可进底稿**——四要素可解释 + 人工复核门闩 + 可治理降级。  
> 请各位评委记住：AuditDraft 赋能项目组做初筛与草稿底稿——**人做判断，系统做规模化执行。**  
> 谢谢，欢迎挑战：可上传你们自己的 CSV。

**鞠躬停顿**，等提问。

## 评委追问速查（路演必备）

| 提问 | 应答要点（≤20 秒） |
| --- | --- |
| 能进项目组吗？ | 定位协审初筛 + draft；须 Review；边界写在 UI 与系统卡 |
| 和 ChatGPT 有何不同？ | 规则保确定性、四要素、Golden Set、RLS、降级；不是对话出意见 |
| 会不会幻觉？ | 硬风险不由 LLM 新增；无证据无结论；可解释性进评测门禁 |
| 误报怎么办？ | 阈值可配置、规则版本化、负例进 Golden Set 回归 |
| 数据安全？ | 演示脱敏；生产 RLS；密钥仅服务端；轨迹可导出 |
| 法律责任？ | 不替代签字；输出 draft；禁止无复核披露 |
| 为何 LangGraph？ | 质量流程编排：可测、可降级、可插人工门闩 |
| F1=1 是否过拟合？ | 演示集有意埋雷验证链路；生产会持续扩负例与边界 case——诚实说「当前是可控演示集上的硬门禁」 |

## 3 分钟压缩版（时间不够时）

| 秒 | 页 | 只说一句 |
| --- | ---: | --- |
| 0–15 | 封面 | 协审工作台，不是聊天写底稿 |
| 15–45 | 问题+定位 | 规模化红旗初筛；规则硬结论，模型只成文 |
| 45–90 | 四要素 | 无证据无结论，才能进复核链 |
| 90–135 | 数字 | Golden Set F1=1；人工 45min vs 系统秒级 |
| 135–165 | 治理 | 降级 + RLS + 人工门闩 + 禁止出意见 |
| 165–180 | 收尾 | 赋能项目组；请看 Demo / 欢迎挑战 CSV |

然后直接 Demo（可砍到 2 分钟：上传 → 风险卡四要素 → `/eval`）。

## 演讲气质建议

1. **像内部提案，不像创业路演**：少形容词，多边界与证据。  
2. **先认边界再谈能力**：评委信任来自「我们知道自己不能做什么」。  
3. **数字只报可复核的**：Precision/Recall/F1、人工对照、行数、埋雷类型。  
4. **Demo 翻车话术提前背**：规则-only / `/demo/snapshot`（见 demo 脚本故障预案）。
