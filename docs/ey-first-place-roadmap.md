# 安永华明 AI 项目赛 · 冲击第一路线图

> 目标：让**技术评委、审计业务评委、风险合规评委、创新/商业评委**对「实用性 / 专业认可度 / 可落地性 / 负责任 AI / 演示说服力」均给出 **9 分以上** 印象，冲击第一名。  
> 原则：本文**不考虑实现难度**，只列方向、功能与优化点；优先级按「对冲第一的边际收益」排序。  
> 语境：本项目参赛主体对标 **四大 · 安永华明（EY）** 的 AI 项目比赛——评委默认用「能否进项目组 / 能否过质量复核 / 能否过信息安全」的尺子打分，而不是黑客松玩具标准。

---

## 0. 评委画像与打分靶心（先对齐尺子）

| 评委类型 | 他们真正在听什么 | 当前短板（相对 9 分） | 冲 9+ 的关键句 |
| --- | --- | --- | --- |
| 审计业务 / 合伙人 | 是否符合审计逻辑、能否进底稿复核链 | 场景窄、数据玩具、缺抽样与重要性 | 「这是费用循环**初筛 + 可复核草稿底稿**，不是替代判断」 |
| 技术 / 数据科学 | 架构是否可扩展、幻觉如何控、评测是否硬 | 缺离线评测集与误报/漏报数字 | 「规则保确定性，LLM 只表达，指标可复现」 |
| 风险 / 合规 / 信安 | 数据出境、权限、审计轨迹、模型治理 | 缺完整 AI 治理与数据分级叙事 | 「EY 级：最小权限、全链路可追溯、可关停」 |
| 创新 / 商业 | 与现有工具差异、可复制到其他业务线 | 叙事偏「又一个 AI 出报告」 | 「可解释混合推理平台，可复制到 AP/AR/Payroll」 |
| 演示现场评委 | 3 分钟内是否「一眼可信」 | 演示集太小、缺对比实验 | 「同表人工 vs 系统：时间与命中对比」 |

**一句话定位（建议全场统一口径）**

> AuditDraft 不是「用大模型写底稿」，而是安永可用的 **费用与报销循环 AI 协审工作台**：  
> **确定性规则引擎**产出可复现发现 → **RAG**锚定准则与方法 → **LLM**只做理解与成文 → **每条结论四要素可解释** → 输出进入 **Prepare → Review → Archive** 复核链。

## 1. 分项冲 9+：必须补齐的「证据包」

> 评委打 9 分以上，靠的不是功能清单，而是**可当场验证的证据**。下列每项都是「没它很难上 9」。

### 1.1 实用性证据（业务评委）

- [ ] **真实量级脱敏数据集**：≥ 2,000～10,000 行费用/报销明细（含正常单 + 边界单 + 舞弊埋雷），字段对齐事务所常见导出（供应商、科目、期间、成本中心、审批链、发票号、支付日期、银行账号后四位等）。
- [ ] **人工基线对比实验**：同一数据集，记录「资深助理人工筛查耗时 / 命中 / 漏报」vs「系统耗时 / 命中 / 漏报 / 误报」，产出一页对比卡（路演必备）。
- [ ] **误报可解释与可调参**：每个规则展示阈值、窗口、金额带；现场可调「审批阈值 / 拆分窗口 / z-score」并即时重跑。
- [ ] **「草稿底稿」边界声明 UI**：明确标注 *AI-assisted draft — subject to engagement review*，避免被质疑「替代注册会计师判断」。
- [ ] **多文件/多期间**：支持一期多表（报销 + 付款流水 + 供应商主数据）勾稽，而不仅是单 CSV。

### 1.2 专业认可度证据（审计方法评委）

- [ ] **对齐安永常见工作方式的语言**：风险点映射到 *significant risk / fraud risk factor / control deficiency* 表述（中英双语可选）。
- [ ] **审计程序链路**：每个 finding 绑定「建议执行的审计程序」（询问、检查、重新执行、分析程序），而不只是风险标签。
- [ ] **重要性 / 明显微小错报**：引入 planning materiality、performance materiality、trivial threshold（可配置），规则与风险评级吃这些参数。
- [ ] **抽样建议模块**：对总体给出推荐抽样方法（货币单位抽样 MUS / 随机 / 判断抽样）与样本量示意（哪怕是简化公式，也要「像审计」）。
- [ ] **工作底稿索引结构**：W/P 编号、交叉引用（`A-1 → A-1-1`）、结论段、例外事项汇总——导出物长得像底稿，而不是博客文章。
- [ ] **复核痕迹（Review Notes）**：Reviewer 可批注「同意 / 需补充证据 / 推翻」，批注进审计轨迹，模拟 EY 二级复核。
- [ ] **准则引用可点开原文片段**：RAG 返回 chunk + 来源文档名 + 条款号；禁止只有书名号没有段落。

### 1.3 技术可信证据（技术评委）

- [ ] **离线评测集（Golden Set）**：固定 50～200 个标注 case（含应命中/不应命中），CI 跑 `precision / recall / F1`，路演甩出数字。
- [ ] **幻觉对抗测试**：故意喂「诱导 LLM 编造风险」的 prompt/数据，证明系统**零无证据结论**（规则层 + guardrail 双保险）。
- [ ] **Provider 热切换实测录像**：OpenAI / Claude / 通义同一任务结论一致性对比（规则 findings 必须 100% 一致；文本表述可不同）。
- [ ] **端到端 SLA**：P50 / P95 耗时仪表（目标：规则秒级；全流程含 LLM 可控在演示 SLA 内），冷启动策略说清。
- [ ] **可观测性**：每个 job 的 node span、token 用量、检索命中、失败原因——像生产系统，不像脚本。

### 1.4 风险合规 / 负责任 AI（信安与质量评委 · EY 极看重）

- [ ] **数据分级与驻留说明**：演示环境数据分级（公开脱敏 / 内部），明确「无真实客户机密」；架构图标注密钥、RLS、service role 边界。
- [ ] **全链路 Audit Trail**：谁上传、谁触发、用了哪版规则/模型/知识库、导出了什么——可导出为合规日志。
- [ ] **模型卡（Model Card）+ 系统卡（System Card）**：能力边界、已知失败模式、禁止用途（不得直接出审计意见）。
- [ ] **Human-in-the-loop 强制点**：高风险 finding 必须人工确认后才能进「最终底稿包」。
- [ ] **Kill switch / 降级模式**：LLM 不可用时，仍可仅跑规则引擎出「发现清单 + 结构化底稿骨架」（现场故障也是加分项）。
- [ ] **提示词与知识库版本管理**：`prompt_version` / `kb_version` / `ruleset_version` 写入每份报告元数据。

### 1.5 创新与可复制（商业/创新评委）

- [ ] **「协审操作系统」叙事**：同一套 graph + rules + RAG，可切换业务包（AP / 费用 / Payroll / 收入截止），现场展示「插件式规则包」。
- [ ] **与黑盒 Copilot 的对照页**：并排「纯 LLM 扫表」vs「AuditDraft」——一侧可能编造，一侧四要素齐全。
- [ ] **安永内部可落地路径一页纸**：试点团队 → 质量复核 → 工具认证 → 推广其他业务线（哪怕是设想，也要像内部提案）。

## 3. 规则与 AI 能力升级清单（专业内核）

### 3.1 规则引擎：从 4 条「演示规则」升级为「费用循环规则包」

建议至少覆盖以下族（每族多条可配置规则）：

1. **重复与重叠**：同发票、同金额近邻、跨科目重复、跨期间重复退费。  
2. **拆单与阈值规避**：时间窗拆分、供应商拆分、项目拆分。  
3. **审批与职责分离**：缺失审批、自审批、审批人=报销人、越权金额带。  
4. **异常金额与分布**：z-score、Benford（首位/前两位）、周末/节假日、整数大额。  
5. **供应商与支付**：新供应商立即大额、银行账号变更后首付、付至个人卡疑似。  
6. **发票与税**：失控发票号段、税率异常、无税号大额（按演示区法规简化）。  
7. **勾稽断裂**：有报销无付款、有付款无报销、金额不一致。  
8. **关联与集中**：同一成本中心异常集中、同一审批人放行异常簇。

每条规则强制输出升级版结构（在现有四要素上扩展）：

```ts
interface AuditFindingV2 {
  riskType: string;
  severity: "low" | "medium" | "high" | "critical";
  triggeredRule: string;
  rulesetVersion: string;
  evidence: unknown;           // 原始行 + 计算字段
  standardRef: string;         // 准则条款
  methodologyRef?: string;     // 程序/方法
  explanation: string;
  recommendedProcedures: string[];
  materialityImpact?: "trivial" | "below_pm" | "above_pm";
  fraudRiskFlag?: boolean;
  reviewerStatus?: "open" | "cleared" | "exception";
}
```

### 3.2 LLM：从「写漂亮」升级为「写得像安永底稿」

- [ ] 强制结构化输出（JSON schema）→ 再渲染为底稿，禁止自由发挥段落游离证据。  
- [ ] 底稿模板分区固定：背景 / 范围 / 程序 / 发现 / 风险评价 / 控制影响 / 建议 / 结论 / 附录证据。  
- [ ] `EXPLAINABILITY_GUARD` 升级为**机器可校验**：生成后校验「每个风险段落必须引用 finding_id」。  
- [ ] 语气：专业、克制、可复核；禁止营销腔与绝对化措辞（「证明舞弊」→「存在需进一步查证的红旗」）。  
- [ ] 提供「合伙人摘要（1 页）」与「执行层详细底稿」两档输出。

### 3.3 RAG：从 15 条种子升级为「可审计知识层」

- [ ] 文档类型分层：准则、监管指引、公开案例、内部程序模板（演示用虚构）、规则说明。  
- [ ] 检索结果 UI：相似度、引用段落、为什么命中。  
- [ ] 「无检索不引用」：没有 chunk 就不写 `standardRef` 具体条款号。  
- [ ] 中英材料与费用审计主题覆盖率清单（评委可抽查）。

### 3.4 Graph：从线性 7 节点升级为「可分支的质量图」

- [ ] 条件边：无高风险可走「简化底稿」；有 critical 强制进入「加强程序建议」。  
- [ ] 人工门闩节点：`awaitHumanReview`（高风险）。  
- [ ] 并行：规则包并行执行 + 汇总。  
- [ ] 每个节点输出写入 span，供 Workflow Tracker 「专业模式」展示输入/输出摘要。

## 4. 产品与 UX：让业务评委「愿意天天用」

- [ ] **工作台信息架构**：Jobs / Findings / Workpapers / Review / Knowledge / Settings（规则与重要性）。  
- [ ] **Finding 工作队列**：筛选 severity、规则、供应商、复核状态；批量清除误报。  
- [ ] **证据抽屉**：点击 finding 右侧滑出原始行、相邻交易、供应商历史。  
- [ ] **一键「复制到复核意见」**、一键「生成询问邮件草稿」（基于证据，可编辑）。  
- [ ] **演示模式开关**：放大字号、高亮四要素、自动播放节点推进（专为路演）。  
- [ ] **无障碍与稳健性**：弱网提示、失败可恢复、导出进度。  
- [ ] **视觉专业度**：克制、事务所工具气质（少炫光多信息密度）；风险色盲友好。

## 5. 叙事、路演与材料（很多第一是「讲赢」的）

### 5.1 必打四张牌（开口顺序建议）

1. **问题**：费用循环例外筛查与底稿记录占用高成本人力。  
2. **方法**：混合推理——规则定责、RAG 锚定、LLM 成文、人复核。  
3. **证明**：对比实验数字 + Golden Set + 现场可复现。  
4. **治理**：版本、轨迹、人工门闩、降级模式——符合 EY 质量与风险偏好。

### 5.2 必备路演资产

- [ ] 3 分钟演示脚本（安永版话术，少用「颠覆」，多用「赋能项目组」）。  
- [ ] 1 页架构图（业务语言版 + 技术版各一）。  
- [ ] 1 页 Responsible AI / 数据安全说明。  
- [ ] 1 页试点方案（范围、成功指标、风险、推广）。  
- [ ] 30 秒故障预案（规则-only 降级演示）。  
- [ ] 「评委提问题库」扩到 30 问（幻觉、法律责任、与现有工具关系、模型选型、成本、偏见、跨境数据等）。

### 5.3 话术红线（避免被业务评委一票否决）

| 不要说 | 改成 |
| --- | --- |
| 替代审计师 / 自动出审计意见 | 辅助项目组初筛与底稿起草，结论需复核 |
| 100% 发现舞弊 | 提升红旗检出效率，降低遗漏，误报可配置 |
| 黑科技 LLM 审计 | 确定性规则 + 生成式表达的混合系统 |
| 已可用于所有审计 | 先聚焦费用/报销循环，方法论可复制 |
